1 Goals & non-goals

What this console is, and just as importantly, what it is not.

In scope

  • Cross-tenant visibility — list and search every organization, user, agency, loan and document on the platform.
  • Lifecycle controls — block / unblock orgs and users at the platform level, with mandatory reason and audit trail.
  • Invite-only onboarding — REAM is closed by default; only invited owners can stand up an organization.
  • Feature flags — a first-class flag store (global / per-org / per-user) controlling, among other things, whether public sign-ups are allowed.
  • Operational instruments — broadcast banners, scheduled-maintenance comms, an immutable audit log, and a system-health dashboard.

Out of scope (v1)

  • Tenant-side admin features — organization owners already have their own admin UI; the platform console is strictly a REAM-internal tool.
  • Direct edits to tenant business data — no editing loans, documents or payments from this console. Read-only with explicit "Open in tenant" links if needed.
  • Billing / metering — handled by the existing billing service. The console links out to Stripe / billing dashboards, doesn't reimplement.
  • Customer support ticketing — Zendesk integration via deep-links; we do not build a help-desk inside the console.

3 Architectural boundaries

Treat the console as a distinct application — separate URL, separate auth realm, separate deploy.

Separate origin

Hosted at admin.ream.com. Different cookie domain than app.ream.com — a compromised tenant session cannot escalate.

Separate auth realm

Identity provider issues platform-only tokens with role claims (platform.super_admin, etc.). Hardware MFA enforced at the IdP level.

IP allow-list

Production console only reachable from the corporate VPN / Cloudflare Access. No public DNS resolution outside the allow-list.

Read replica first

Cross-tenant reads hit a dedicated replica to avoid noisy queries on the OLTP primary. Mutations go through the same service layer as the tenant app.

Append-only audit

Every privileged action writes to platform_audit_log in a separate database with WORM retention (7 years).

Same code, separate deploy

Console is a separate Next.js app in the same monorepo. Shares the domain libs, has its own CI/CD pipeline and feature-flagged build.

5 Feature inventory

The console split into ten functional modules. MUST are required for v1; SHOULD for GA; MAY are stretch.

Tenants MUST

Cross-tenant organization directory. Filter by status, plan, region, signup source. Drill-in: users, agencies, loans, documents, billing link, audit slice.

Users MUST

Global directory across tenants. Search by email, phone, ID. Block/unblock at platform level (overrides tenant). View login history.

Invitations MUST

Generate single-use or multi-use invite links. Set expiry, intended role, allowed email domain. Revoke, resend, copy link, bulk CSV.

Feature flags MUST

Boolean / string / number / JSON flags. Global default, per-org override, per-user override, percentage rollout. Change history per flag.

Block / unblock MUST

Three-state suspension (Active → Read-only → Locked) with mandatory reason code & note. Auto-expire option for temp holds.

Audit log MUST

Searchable, exportable, immutable. Every console action is recorded with before/after JSON snapshot, IP and UA.

Broadcasts SHOULD

System banners across the tenant app — incident, planned maintenance, release notes. Severity-coloured, scheduled, dismissable.

Health dashboard SHOULD

SLO uptime, error budget burn, queue depth, top slow endpoints. Embedded Grafana / Datadog widgets, no original metrics work.

Kill-switch SHOULD

One-click read-only mode for a tenant or the whole platform — wired through the flag system. 2-person approval at platform-wide level.

Data export MAY

GDPR/CCPA right-of-access — export all data for an org or user as a signed ZIP. Sensitive: requires step-up MFA + reason note.

6 Invite-gated onboarding

REAM is a closed platform. Owners can only register an organization through an invitation, unless a super admin explicitly opens the door.

Invite token shape

{
  "token":        "inv_01HNQT8G3F4Z5ABXDR9V0K2T7M",
  "email":        "founder@acmerealty.com",
  "intended_role": "org_owner",
  "allowed_domain": "acmerealty.com",    // optional
  "max_uses":     1,
  "used_count":   0,
  "expires_at":   "2026-06-01T00:00:00Z",
  "created_by":   "padmin_42",
  "status":       "active"          // active | revoked | exhausted | expired
}

Sign-up gate logic

// Server-side org creation handler
async function registerOrg(req) {
  const publicSignups = await flags.eval("signups.public_enabled");

  if (!publicSignups) {
    const token = req.body.invite_token;
    if (!token) throw Forbidden("invite_required");

    const invite = await invites.findActive(token);
    if (!invite)               throw Forbidden("invite_invalid");
    if (invite.expires_at < now()) throw Forbidden("invite_expired");
    if (invite.used_count >= invite.max_uses) throw Forbidden("invite_exhausted");
    if (invite.email && invite.email !== req.body.email)
                              throw Forbidden("invite_email_mismatch");

    await invites.consume(invite.id, req.user.id);
  }

  return orgs.create({ ...req.body, owner_id: req.user.id });
}

UX flow

  1. Operator generates an invite from /admin/invites — paste email, pick max-uses (default 1), pick expiry (default 14 days), copy the link.
  2. Recipient lands on app.ream.com/register?invite=<token>. The invite is pre-validated; the email is pre-filled and locked if the invite is email-bound.
  3. On submit, the server re-validates and consumes the token in the same transaction as orgs.create() — no race.
  4. The console shows the invite as used, with the resulting org and user IDs linked.
Edge cases to handle. Token reuse attempts, email-bound invites being claimed by a different signed-in user, expired tokens, the invite-link being shared on social media (rate-limit + audit), and the case where a user already has an account at another org and tries to redeem an invite for a new one.

Resolution order

  1. User-scoped override (if any) →
  2. Org-scoped override (if any) →
  3. Percentage rollout cohort (if defined; bucketed by stable hash of org_id) →
  4. Global value →
  5. Compiled-in default.
Build vs buy. Use OpenFeature as the API surface from day one. Back it with our own table for v1, and swap to LaunchDarkly / Flagsmith later without touching call-sites if scale demands it.

8 Block / unblock semantics

Three states, mandatory reason codes, and a clear contract about what each state allows.

StateOrg behaviorUser behaviorTypical reason
ActiveNormal operation.Normal operation.—
Read-onlyCan sign in & read; all mutations return 403 read_only.Can sign in & read everything they had access to. Cannot upload, pay, sign.Payment failure, KYC re-review, customer dispute.
LockedCannot sign in. Org dashboard shows the reason and a contact link.Cannot sign in. Sessions revoked.Fraud, abuse, court order, term violation.

Workflow

  1. Operator opens an org or user record, clicks Suspend.
  2. Modal requires: target state, reason code (enum), free-text note (≥20 chars), optional auto-expire.
  3. Step-up MFA prompt before commit.
  4. On commit: state row written, all active sessions revoked (Locked) or marked read-only (Read-only), org.suspended webhook fired, audit row written, customer-side banner shown on next session.
  5. Unblock follows the same flow with reason code cleared and an additional resolution note.

9 Audit, compliance & security

  • Every privileged action writes a row to platform_audit_log. Schema includes actor, role, action, target, before/after JSON snapshot, IP, user-agent, request-id and timestamp.
  • Step-up MFA on destructive actions (block, kill-switch, data export, role assignment) — re-prompt for hardware key even if recently authenticated.
  • 2-person rule on platform-wide kill-switch and on data export: a second admin must approve within 15 minutes.
  • Reason-required — every mutation has a reason field; nullable reasons fail validation. The reason flows into audit and into the customer-facing message where applicable.
  • PII discipline — never log SSN, full card PAN, or signature images. Document references are linked, not embedded.
  • Retention — audit log is WORM with 7-year retention; tenant data export is encrypted at rest with a per-export KMS key and auto-deleted after 30 days.
  • Compliance hooks — GDPR Article 15 (right of access), Article 17 (right of erasure), CCPA §1798.105, SOC 2 CC7.2 (system monitoring), CC6.1 (logical access).
Threat model worth naming. The biggest risk is a compromised platform-admin session, not an external attacker. Therefore: short session lifetimes (4 h), continuous re-auth on sensitive screens, IP allow-list, hardware MFA, and Slack-alerting on every high-severity action.

11 Phased rollout — 14 weeks

Cut at every line — each phase ends with something usable in production by REAM operators.

Phase 0Weeks 1–2Foundation

Foundation & infra

Stand up admin.ream.com, separate auth realm, hardware MFA enforcement, Cloudflare Access IP allow-list, audit-log table, OpenFeature client wired into the tenant app.

  • Create platform_admins, feature_flags, flag_evaluations, platform_audit_log tables.
  • Bootstrap the first super admin via DB migration; that admin invites the rest in-product.
Phase 1Weeks 3–5Read-only

Read-only console

Cross-tenant directories, drill-ins, audit search, dashboard KPIs. No mutations yet — proves the read replica, the role guard and the audit pipeline.

  • Screens: /admin/dashboard, /admin/orgs, /admin/orgs/:id, /admin/users, /admin/users/:id, /admin/audit.
  • Exit criteria: ops team uses the console for every customer-support inquiry instead of running ad-hoc SQL.
Phase 2Weeks 6–7Invitations

Invitation system

The invite store, generation UI, validation in orgs.create(). Public sign-up is gated by the new signups.public_enabled flag (defaulting to false).

  • Screens: /admin/invites, plus an updated /register on the tenant app.
  • Backfill: every existing org gets a synthetic "grandfathered" invite record so the cohort is consistent.
Phase 3Weeks 8–9Flags

Feature-flag platform

Full flag editor: global value, per-org and per-user overrides, percentage rollout, change history. signups.public_enabled moves from hard-coded to the new editor.

  • Screens: /admin/flags + per-flag detail.
  • SDK: thin wrapper over OpenFeature exposed to backend services and the SPA.
Phase 4Week 10Block / unblock

Lifecycle controls

Three-state suspension with reason codes, step-up MFA, session revocation, customer-facing banners.

  • Hooks: API gateway respects org.status = read_only|locked on every request.
  • Webhook: org.suspended, org.restored, user.suspended, user.restored.
Phase 5Week 11Comms

Broadcasts & health

System banner editor, scheduled maintenance windows, embedded health dashboard.

  • Screens: /admin/broadcasts, /admin/health.
  • Tenant SDK exposes useBroadcasts() for the existing app shell.
Phase 6Weeks 12–13SSO + RBAC

SSO & granular RBAC

Move platform auth to corporate SSO (Okta / Azure AD). Implement the three-role matrix; add the kill-switch with 2-person rule.

  • Screen: /admin/settings.
Phase 7Week 14Hardening

GA hardening

Pen test, threat model review, SOC 2 evidence collection, runbook for incident-response, on-call training. Console marked GA.

Capacity assumptions. 1 backend engineer + 1 frontend engineer + 0.25 designer + 0.25 security. Phase boundaries are soft — Phase 1 reads can be shipped independently while Phase 2's invite work proceeds in parallel after the foundation is in place.

12 Open questions

  • Single region or multi-region from day one? Affects whether the audit log is centralised or per-region with replication.
  • Build flags or buy? OpenFeature gives us optionality; we still need to commit to a back-end for v1. Recommendation: ship our own table for control + cost, design the schema so a LaunchDarkly migration is a one-day swap.
  • Org owner identity reuse. If a user already exists at Org A and is invited to Org B, do they get a second account or a multi-tenant identity? The data model assumes the latter; the UX needs design.
  • Public-signups landing page. When the flag flips on, what does app.ream.com/register look like? Drafted in a follow-up doc.
  • Customer notification on block. Is an email always sent? Required for read-only? For locked? Compliance has a view here.
  • Admin console look-and-feel. Reuse the existing REAM design system, or differentiate (different brand colour, "internal" stripe) so screenshots can never be confused with the customer app?
